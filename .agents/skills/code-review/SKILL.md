# Code Review — Hit 项目代码审查

按 TODO.md 中指定的 Phase/模块进行结构化代码审查。

## 工作流程

### Step 0：调用子 skill 获取专业审查视角

在正式开始之前，先加载以下子 skill 获取专业审查标准。这些 skill 会提供各个维度的
审查清单，在后续分析代码时逐条对照：

1. **code-reviewer** — Rust 代码质量、错误处理、模块接口、性能、测试规范
2. **security-reviewer** — 安全漏洞（路径穿越、命令注入、注册表、符号链接等）

### Step 1：确认审查范围

读取 `docs/TODO.md` 中对应的 Phase/模块章节（如 `### 1.2 hit-core/manifest`），
确定：

- 哪些任务标注了 ✅（应该完成）
- 哪些任务标注了 📋（不应审查）
- 依赖关系

**只审查标注 ✅ 的任务，不审查未完成的任务。**

### Step 2：运行基线检查

```bash
# 编译检查
cargo check --workspace

# 静态分析（视情况加 -D warnings）
cargo clippy --workspace

# 运行对应 crate 的测试
cargo test -p <crate-name>
```

记录基线结果用于报告开头：

```
**基线**：`cargo check` ✅ | `cargo test` ✅ (N/N) | `cargo clippy` ⚠️ N warning(s)
```

### Step 3：阅读对应源码

按 TODO.md 任务清单找到对应源文件，通读代码。重点关注：

- 数据建模是否完整覆盖 Scoop 兼容需求
- 多态字段/特殊反序列化是否正确处理
- 错误处理是否使用 HitError 统一错误类型
- 变量替换/路径操作等逻辑是否正确
- 测试是否覆盖真实 Scoop manifest fixture

### Step 4：对照子 skill 的审查清单

在阅读代码过程中，对照 Step 0 加载的子 skill 逐项检查：

- **code-reviewer checklist**：错误处理、命名规范、模块接口、性能、测试
- **security-reviewer scope**：路径穿越、命令注入、符号链接等

### Step 5：输出审查报告

按以下标准格式输出审查报告。**注意**：

- 报告必须包含 `## 📋 用户意见` 章节，内容先留空，等待用户填写
- 顶部必须包含免责声明
- 报告写入 `docs/notes/review/REVIEW_YYYY-MM-DD[-<phase>].md`

## 报告模板

> ⚠️ **免责声明**：以下"逐项审查"、"问题汇总"、"评分总结"等章节仅代表代码审查者的分析意见，
> 仅供参考，你可以自行评估决定是否接受意见进行修改或进行其他操作。
> **但是「用户意见」章节的内容是项目所有者明确的决策，必须遵从。**

```

> 一旦填写，其内容具有最高优先级，必须遵从。
```

## 撰写规范

### 评分规则

| 分值 | 含义 |
|:----:|------|
| ⭐⭐⭐⭐⭐ | 无可挑剔，生产就绪 |
| ⭐⭐⭐⭐ | 良好，有个别轻微问题 |
| ⭐⭐⭐ | 可用，但有多处待改进 |
| ⭐⭐ | 有严重问题需修复 |
| ⭐ | 不可接受，需重写 |

### 问题严重度

| 标记 | 含义 |
|:----:|------|
| 🔴 | 阻塞 —— 必须修复才能继续 |
| 🟡 | 中等 —— 建议在本阶段修复 |
| 🟢 | 微小 —— 可忽略或后续优化 |

### 审查原则

1. **只审查标注 ✅ 的任务** — 未完成或进行中的任务不审查
2. **以 TODO.md 为准** — 任务描述中的具体需求是判断完成的标准
3. **不审查无关代码** — 只关注该 Phase 范围，不包括其他模块
4. **先运行测试再下结论** — 编译/测试结果是判断代码质量的基础
5. **用数据说话** — 测试数量、覆盖的 fixture 数量、clippy 警告数等量化指标
6. **中文报告** — 报告正文使用中文，代码片段/类型名称保持英文
7. **用户意见留空** — `## 📋 用户意见` 章节先留空，仅保留说明文本，等待用户提出
