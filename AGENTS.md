# AGENTS.md

This file provides guidance to the AI agent when working with code in this repository.

## 项目概述

Hit 是一个用 Rust 编写的现代 Windows 包管理器，完全兼容 Scoop 的 bucket/manifest 生态。首要约束是 **Scoop 兼容性** —— manifest JSON 格式、bucket 目录布局和 shim/persist 机制必须与现有 Scoop bucket 兼容。

## 工作区结构

项目为 Cargo workspace，目前包含 5 个 crate 位于 `crates/` 目录下。新建 crate 遵循 `hit-<name>` 命名规范（如 `hit-cli`、`hit-core`）。

## Windows 特有注意事项

- 涉及 UAC 提升、注册表操作、PATH 管理和 `WM_SETTINGCHANGE` 广播
- 使用 `windows` / `winreg` crate 访问 Windows API
- **PowerShell 版本**：本项目使用 PowerShell 7（`pwsh`），不要用 `powershell`（5 版本）。执行 PowerShell 命令时一律用 `pwsh`

## 开发约定

- **注释语言**：代码注释使用中文
- **分支策略**：直接在 `main` 分支开发，不使用 feature 分支
- **Rust 版本**：edition 2024，注意与 2021 的差异
- 每次编辑 `.rs` 文件后会自动运行 `cargo check`（已有 hook）
- 请勿手动编辑 `Cargo.lock`，使用 `cargo update` 更新依赖

> **权衡考量**：本指南优先考虑谨慎性而非速度。对于简单任务，请自行判断灵活处理。

## 1. 三思而后行

不主观臆断、不回避困惑、清晰呈现利弊。

开始前需确认：

- 明确陈述你的假设；不确定时务必询问
- 存在多种解读时，应主动提出而非暗自选择
- 有更优方案时，请说明；必要时可提出异议
- 遇到疑问，立即暂停，明确指出问题并寻求澄清

## 2. 简洁至上

用最少代码解决问题，不做多余推测。

- 不添加超出需求的功能
- 不为一次性代码设计抽象层
- 不引入未被要求的"灵活性"或"可配置性"
- 不为不可能发生的场景编写错误处理
- 能用50行解决的问题绝不写200行
- 时常自问："资深工程师会觉得这过于复杂吗？"若是，立即简化

## 3. 精准改动

只修改必须改的部分，只清理自己引入的冗余。

编辑现有代码时：

- 不"优化"无关的代码、注释或格式
- 不重构运行正常的代码
- 遵循现有代码风格，即便你有不同偏好
- 发现无关死代码可提及，但不要擅自删除

当你的改动产生孤立代码时：

- 移除因你的改动而不再使用的导入、变量或函数
- 除非明确要求，否则不删除已存在的死代码

**检验标准**：每一处改动都应能直接追溯到用户的需求。

## 4. 目标导向执行

定义明确的成功标准，循环验证直至达成。

将任务转化为可验证的目标：

| 任务       | 验证目标                     |
| :--------- | :--------------------------- |
| "添加验证" | 为无效输入编写测试并使其通过 |
| "修复bug"  | 编写复现测试用例并使其通过   |
| "重构X"    | 确保重构前后测试全部通过     |

多步骤任务需制定简要计划：

1. [步骤] → 验证：[检查项]
2. [步骤] → 验证：[检查项]
3. [步骤] → 验证：[检查项]

明确的成功标准能让你独立推进工作；模糊的标准（如"让它能运行"）则会导致反复澄清。

## 5. 代码规范

- 代码应当添加适当的注释，包括：
  - 函数和方法的用途说明
  - 复杂逻辑的解释
  - 关键变量和常量的含义
  - 代码变更的原因
- 保持代码风格一致，遵循项目现有的代码规范
- 代码结构清晰，避免过度嵌套和复杂的单行代码

> 注释是一个好习惯，但也应当避免过度注释，过于简单的逻辑不需要注释。
