## 🔄 文档流转流程

```
新想法
   │
   ▼
┌─────────────────┐
│  EUREKA.md       │  ← 随手记录，一句话描述
│  (灵感池)        │
└────────┬────────┘
         │ 确定要实施
         ▼
┌─────────────────┐
│  PROJECT.md      │  ← 写入正式项目描述（功能、结构）
│  (项目描述)       │
└────────┬────────┘
         │ 进入开发
         ▼
┌─────────────────┐
│  TODO.md         │  ← 拆解为具体任务，逐项追踪
│  (任务清单)       │     实施中更新状态 ✅ / 📋
└────────┬────────┘
         │ 完成一阶段
         ▼
┌─────────────────┐
│  REVIEW_*.md     │  ← 代码审查，记录发现与决策
│  (审查报告)       │
└─────────────────┘
```

**简要说明**：

1. **新想法诞生** → 先写入 `EUREKA.md`，一句话记录核心思路，附带评审意见
2. **确定要实施** → 在 `PROJECT.md` 中补充功能描述（如果涉及新功能）
3. **进入开发** → 在 `TODO.md` 中拆解为具体任务，标注状态
4. **完成审查** → 审查报告写入 `review/`，发现的决策意见回退到 `TODO.md` 或 `PROJECT.md`

### 规则

- **EUREKA 是最宽松的**：任何想法都可以写，不需要成熟度
- **PROJECT 是正式描述**：内容可迭代，但需保持与当前实现一致
- **TODO 是唯一权威**：Phase 1-3 范围内，以 TODO.md 为准。当其他文档与 TODO 冲突时，以 TODO 为准
- **文件之间通过相对路径互相引用**，避免信息孤岛
